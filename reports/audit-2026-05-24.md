# Acts audit — 2026-05-24

Read-only health check. No data modified.

## Focus

Cards surfacing under the **"5 Minutes Max"** filter chip but which clearly take longer than 5 minutes (in-person protests, multi-hour events, etc.).

## How the filter actually works

The "5 Minutes Max" chip in the navbar toggles `quickActionsOnly`. App.tsx then filters via:

```ts
if (quickActionsOnly && !card.quickAction) return false;
```

So the filter shows **only cards with `quickAction: true`**. It does NOT look at `timeCommitment`. That means any card flagged `quickAction: true` shows up under "5 Minutes Max" regardless of what its timeCommitment string says or how long it actually takes.

## ⚠️ Root cause of the specific cards Ellen saw — pin band bypasses the filter

The two cards Ellen saw on screen — **#2159 (NO WAR. NO KINGS: U Village — Seattle)** and **#2160 (Protest ICE Terror at ICE HQ — Tukwila WA)** — both have `quickAction: undefined` and `category: PROTEST`. They should NOT pass the `quickActionsOnly` filter.

They're slipping through via a separate code path: the **state-local upcoming events pin band** added in v1.1.50 at `App.tsx:587-619`. That loop lifts any card that has a future `eventDate` and matches the user's state (set via Match Me). It checks:

1. ✓ future `eventDate`
2. ✓ `cardIsLocalToState(c, userState)`
3. ✓ `settingMatches(c, userSetting)` — respects Remote vs In-Person
4. ✓ `adminApproved` (or admin viewer)

But it does NOT check:

- ✗ `quickActionsOnly` filter
- ✗ Category/location filter chips
- ✗ Search query

Ellen's Match Me state is Washington; both cards are Washington with eventDate of May 29/30; both bypass "5 Minutes Max" and appear at the top of the matched feed.

**Suggested fix:** either (a) make the pin band honor `quickActionsOnly` and `activeFilters` — which preserves the pin's "state-local events surface high" intent without contradicting filter chips, or (b) drop the pin band entirely and rely on the matcher's `eventProximityBonus` + `stateBonus` to rank these naturally. (a) is the smaller change.

All of these should have `quickAction: false` (or the field removed). Their `timeCommitment` strings should also be reviewed:

- **PROTEST / FLASH MOB / MEETING / EVENT** → `timeCommitment: "1–3 hours"`, `quickAction: false`
- **CRAFTING / ART PIECE / PROFESSIONAL SKILLS** → `timeCommitment: "1–3 hours"`, `quickAction: false`
- **TRAINING** → `timeCommitment: "1–3 hours"`, `quickAction: false`
- **JOIN A GROUP / LABOR** → `timeCommitment: "Ongoing"`, `quickAction: false`

## How to batch-fix

Two options:

1. **Admin panel** — open each card, uncheck "Quick action", set Time Commitment dropdown, Save. Slowest but no code changes.
2. **One-time KV migration** in the edge function targeting these specific IDs. Gate behind `migration:fix-quickaction-mistags:v1`. Sample skeleton in the codebase: see other `migration:*` blocks at the top of `/actions`. This is the right call if the list grows past ~10 cards.
